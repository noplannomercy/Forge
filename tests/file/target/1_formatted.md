현대케피코 LLM 거버넌스 제안서 (v0.1)

목표: 폐쇄망 환경에서 OpenWebUI + LiteLLM + vLLM + n8n 기반 LLM 서비스를 안전·효율적으로 운영하고, 2단계로 *바이브코딩(클라인/Cline 또는 RooCode + VSCode + 로컬코딩모델)*까지 확장한다.

0. 범위·가정·비범위

범위: 대화형/질의응답형 UI, 모델 프록시, 추론 서버, 오케스트레이션, 로깅/감사, 코드에이전트(바이브코딩).

가정: 인터넷 차단 또는 제한, USB/사내 레지스트리로 아티팩트 반입, 중앙 인증(SSO) 또는 로컬 계정 병행.

비범위: 퍼블릭 클라우드 API 사용, 멀티테넌트 외부 사용자.

1. 참조 아키텍처

1.1 단계 1: 폐쇄망 LLM 서비스

UI: OpenWebUI(워크스페이스/프로젝트 단위 권한)

모델 프록시: LiteLLM(라우팅·쿼터·레이트리밋·감사)

추론: vLLM(로컬 모델: Gemma3 27B, Qwen2.5/QwenCode 등)

오케스트레이션: n8n(전·후처리, 가드레일, 로그 수집)

저장: 로그/메트릭(ES/ClickHouse 등), 정책/메타(PostgreSQL)

RAG(옵션): 임베딩(한국어 특화) + pgvector(Supabase 호환 배포)

요청 흐름

OpenWebUI → LiteLLM(정책/라우팅) → n8n 필터(PII/비밀 제거) → vLLM 추론 → n8n 후처리(출처/라벨링) → 응답 + 로그/비용 기록

1.2 단계 2: 바이브코딩 환경

개발 도구: VSCode + (Cline 또는 RooCode)

모델: Gemma3 27B-Instruct, QwenCode 계열(로컬/vLLM)

실행 보호: 샌드박스(컨테이너/Devcontainer), 읽기우선 모드, PR 생성 중심 워크플로우

감사: 에이전트 명령·파일변경 diff·터미널 실행 로그 수집/서명

2. 거버넌스 원칙(PRISM)

Privacy: PII/비밀 자동 마스킹·차단(입·출력).

Risk-first: 모델/파이프라인 변경 전 사전 위험평가.

Integrity: 소스·모델·정책 버전 고정, 변경 이력.

Security: 최소권한·네트워크 분리·서명된 아티팩트 반입.

Measurability: 품질/비용/보안 로그 일관 수집.

3. 정책(요약)

3.1 수용 가능한 사용(AUP)

주민번호/계좌/비밀번호/액세스 토큰 등 입력 금지, 감지 시 차단.

소스코드 업로드는 사전 승인 리포지토리·경로만 허용.

3.2 데이터 분류·메타

등급: Public / Internal / Confidential / Restricted(PII/법규)

메타 필수: owner, data_class, retention, legal_basis, source_id

3.3 모델 생명주기(온보딩→운영→폐기)

온보딩: 모델 카드 → 보안·법무 리뷰 → 샌드박스 평가 → 승인

운영: SLO(지연/정확도/환각률) 모니터링·월간 리포트

폐기: 토큰 비활성·아티팩트 봉인·로그 보존

3.4 공급망/반입

USB/레지스트리 반입 시 SHA-256 해시·SBOM 스캔·서명 검증.

컨테이너 이미지: Private Registry, 태그+다이제스트 고정.

4. 통제(Controls) 카탈로그

접근제어: OpenWebUI 워크스페이스 RBAC, LiteLLM 프로젝트별 키, vLLM 노드 접근 방화벽.

네트워크: egress 화이트리스트, 공용망 차단, 레지스트리/라이선스 서버만 허용.

비밀관리: Vault/KMS, 주기 로테이션, 환경변수 주입 금지(파일/Agent 정책).

프롬프트 가드: 금칙어·패턴 필터, 토큰/길이 제한, 시스템 프로ンプ트 버전 관리.

출처/서명: 응답 provenance(문서ID, 스니펫, 모델/정책 버전) 표기.

관측성: ts, user, project, model, route, prompt_hash, pii_flags, tokens, latency, cost, decision, agent_cmds[] 스키마.

비용/성능: 사용량 한도, 레이트리밋, P95 지연·토큰/초 모니터링.

5. 바이브코딩(에이전트) 특화 정책

권한 기본값: 읽기 전용 → PR 생성만 허용(메인 브랜치 직접 푸시는 금지).

명령 가드: 파일 시스템/네트워크/프로세스 실행은 허용 리스트 기반.

경로 스코핑: /workspace/app, /workspace/docs 등 한정, OS 전역 접근 금지.

비밀 차단: .env, 키스토어, SSH 키 접근 금지·스캔.

패키지 고정: 오프라인 미러+잠금파일, 서명 검증.

로그: 프롬프트/명령/파일변경 diff/터미널 출력 전량 수집, 변경 서명.

승인 게이트: 위험 변경(보안/빌드/배포 스크립트)은 리뷰어 2인 승인.

6. SOP(표준 절차)

6.1 모델 온보딩 체크리스트

모델 카드(데이터/라이선스/제약/성능) 제출

샌드박스 평가: 환각/독성/PII 유출/코드보안 점수

성능·비용 SLO 합의(예: P95 ≤ 2.5s, 환각 <3%)

운영 승인·버전 고정(모델/토크나이저/텐서 병렬)

6.2 운영/모니터링

LiteLLM 사용량/오류율/레이턴시 대시보드

vLLM 노드 헬스/메모리/throughput

n8n 실패 재시도·데드레터 큐

월간 품질 리포트(정확도, 거절율, 보안 이벤트)

6.3 변경관리

이미지/모델 교체는 변경요청서(CR) + 롤백 플랜 필수

해시·서명·SBOM 증적 보관, 패치 윈도우 운영

6.4 사고 대응

PII/비밀 유출 탐지 시 즉시 차단, 세션 격리, 포렌식 로그 수집

24h 이내 RCA/재발방지 제출

7. KPI / 수용 기준

보안: 금칙/PII 탐지 차단율 ≥ 99%, 무승인 egress 0건.

성능: 사용자 P95 지연 ≤ 2.5s(텍스트), 코드 생성 200 tokens/s 노드당.

품질: 업무 정답률/코드 테스트 통과율 목표치 달성(도메인 평가 셋).

운영: 가용성 99.9%, 미해결 인시던트 0건/분기.

12. 비용 산정 기준(초안)

통화/기준

통화: KRW(부가세 별도). 기준 시점: 2025 Q3. 사내 레지스트리/오프라인 반입 전제.

소프트웨어: OpenWebUI(OSS), LiteLLM(OSS), vLLM(OSS), n8n(OSS/Fair-code) → 라이선스 비용 0원 가정(엔터프라이즈 기능 제외).

하드웨어 단가는 내부 가정치로 범위 제공(벤더 견적 수령 후 확정).

단가 가정(범위)

옵션 A: L40S 48GB ×2 장 GPU 서버: ₩35M ~ ₩55M (CPU/RAM/스토리지 포함 섀시 2U 기준)

옵션 B: H100 80GB ×1 장 GPU 서버: ₩80M ~ ₩140M

AI Server(연계): ₩3M ~ ₩6M

25GbE 스위치/옵틱(필요 시): ₩3M ~ ₩8M

스토리지 증설(NVMe 2TB×2): ₩0.2M ~ ₩0.4M

랙/전원 부자재: ₩0.5M ~ ₩1.0M

구축 인건비: 기본 510 MD / 보안·로그 고도화 35 MD (MD 단가는 내부 기준 적용)

산정 방식

CAPEX = (GPU 서버 단가 × 수량) + (AI Server × 수량) + 네트워크/스토리지/부자재 + 구축 MD×단가

OPEX(전력) = 장비 kW × 평균가동률 × 720h × 전력단가 × PUE

전력단가 가정: ₩180/kWh, PUE: 1.5, 평균가동률: 60%

14. OPEX(전력/유지보수) 추정

전력 가정 계산

옵션 A(L40S×2): 1.5kW 명목 × 60% × 720h × ₩180 × PUE 1.5 ≈ ₩175k/월 (연간 ₩2.1M)

옵션 B(H100×1): 3.0kW 명목 × 60% × 720h × ₩180 × PUE 1.5 ≈ ₩350k/월 (연간 ₩4.2M)

유지보수

하드웨어 보증 3년(권장) / 연장 시 연 5~12%(장비가 기준) 가정.

소프트웨어는 OSS 운영: 보안패치·업데이트는 월간 패치 윈도우에 포함.

17.5 적용 방법

실제 업무 평균 완성 토큰과 요청 빈도를 한번만 계측해 위 표의 시나리오를 보정

5명 초과 시 요구TPS = (인원 × 1인TPM_out) / 60으로 선형 계산

PoC에서 P95 지연과 토큰/초를 실측 → 11장 권고 스펙/13장 견적 확정

필요하면 5→10→20명 스케일 표도 바로 추가하겠습니다.

18. 권고안(H100 1장) 선정 기준 및 근거

본 권고는 **문서 내 가정(11장, 17장)**을 기반으로 한 의사결정 기준입니다. 실제 구매 전 PoC 실측으로 확인합니다.

18.1 운영 목표/제약 (기준값)

사용 패턴: 5명, S3 고부하 시나리오(12 req/h, 완성 1,600 토큰) 발생 가능

지연 목표: 사용자 체감 P95 ≤ 5~7초 (바이브코딩/리뷰 대기 허용 한계)

모델/컨텍스트: 27B급 모델, 8k~16k 컨텍스트 수요 가능

확장성: 612개월 내 1015명까지 확장 가능성 고려

18.2 기술적 근거

처리량·지연: 17.2표 기준, S3에서 H100은 ~6.9s, L40S×2는 ~13.8s → 목표 P95(≤7s)를 안정적으로 충족하는 쪽은 H100

배치/컨텍스트 여유: H100 80GB는 vLLM의 KV-Cache/배칭에 여유가 커 8k~16k 컨텍스트와 동시세션에서 페이징/오프로딩 위험이 낮음

운영 단순성: 단일 GPU로 텐서 병렬 불필요, 모델 배포·모니터링·롤백이 단순

성장 여지: 250 TPS 가정으로 10~15명 확장 및 장문 생성(코드/리뷰)에도 헤드룸 확보

18.3 비용/효율 관점

CAPEX는 높지만, 고부하 시나리오에서 응답시간 절반 수준으로 사용자 생산성/경험 향상

TPS당 원가(중간값 기준)는 A/B가 유사한 범위로, 지연 목표 달성 여부가 차별점

18.4 예외—L40S×2를 선택하는 조건

지연 목표가 느슨(P95 ≤ 10~12s 허용)하거나, 예산 최우선

가용성을 중시(장애 시 단일 GPU 리스크 회피) → 2장 구성이 유리

컨텍스트 길이가 짧고(≤8k), 주 사용이 S1/S2 위주

18.5 의사결정 요약(체크리스트)

P95 목표 ≤ 7s → H100 권고

8k~16k 컨텍스트 + 동시세션 필요 → H100 권고

612개월 내 1015명 확장 → H100 권고

예산 우선 + P95 10s 허용 + 가용성 중시 → L40S×2 가능
